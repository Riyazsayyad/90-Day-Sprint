Treeification

~7 minutes · builds on resize & load factor

The problem: O(n) worst-case

In standard hash tables, if many keys hash to the same bucket (due to poor hashCode() or an intentional hash collision attack), search times degrade from O(1) to O(n). In Java 8, the JDK developers addressed this by introducing treeification.

Source: Oracle HashMap — Implementation Notes

Red-Black Trees

When a bucket's collision chain grows too long, the linked list is converted into a self-balancing Red-Black Tree. Under high collisions, this changes lookup and insertion performance from O(n) to O(log n).

Instead of Node instances, the bucket uses TreeNode instances (which hold left, right, parent, and red properties).

Linked List (O(n)): [Bucket 2] → A → B → C → D → E → F → G → H → I (9 nodes) Red-Black Tree (O(log n)): [Bucket 2] → E / \ B G / \ / \ A C F H \ \ D I

The two conditions for treeification

A bucket does not instantly treeify just because it has collisions. Two thresholds must be met:

  1. Treeify Threshold (TREEIFY_THRESHOLD = 8): A linked list chain must grow to length 8. When adding the 9th node (chain length becomes 9), treeification is triggered.
  2. Minimum Capacity (MIN_TREEIFY_CAPACITY = 64): The total capacity of the map must be at least 64. If the map is smaller than 64 (e.g. default capacity of 16), the HashMap will trigger a resize() instead of treeifying. Resizing splits the bucket and distributes nodes, which usually resolves the collision chain.

Source: JDK 8+ HashMap.java source code (treeifyBin method)

Untreeification (The Hysteresis)

If the number of entries in a treeified bucket decreases (due to removals or split buckets during resizing), the tree is converted back to a linked list. This happens when the count falls to UNTREEIFY_THRESHOLD = 6.

Why 6 and not 8? Having different thresholds (8 for treeifying, 6 for untreeifying) prevents thrashing — rapid, expensive conversions back and forth between a list and a tree if the bucket size fluctuates around 8.
