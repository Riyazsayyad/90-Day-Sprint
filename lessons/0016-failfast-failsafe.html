Java Fail-Fast vs. Fail-Safe Iterators: Interview Guide

~8 minutes · ties to mission

1. What is ConcurrentModificationException (CME)?

In Java, iterating over a collection while modifying its structure (adding, removing, or clearing elements) is generally unsafe. If the runtime detects an illegal structural modification during traversal, it throws a ConcurrentModificationException.

2. modCount and expectedModCount Internals

Standard collections (like ArrayList) track modifications internally. The fail-fast iterator uses two fields to enforce safety:

  • modCount: An index variable on the collection class. It increments on every add, remove, or clear structure call.
  • expectedModCount: A local cached copy stored inside the Iterator object upon creation.

In every call to next() or remove(), the iterator calls checkForComodification():

final void checkForComodification() {
    if (modCount != expectedModCount)
        throw new ConcurrentModificationException();
}

3. Why iterator.remove() updates modCount correctly

If you call list.remove() during iteration, only modCount changes, causing a subsequent next() call to detect a mismatch and crash.
However, calling iterator.remove() is safe because the iterator deletes the element through the collection, increments the collection's modCount, and then synchronizes its cached state: 

expectedModCount = modCount; // Re-synchronized!

4. Why Fail-Fast is Best-Effort

The Javadoc states that fail-fast behavior is **best-effort** and **not guaranteed**. Here is why:

  • Visibility Gaps: In a multithreaded context, modCount is not declared volatile or synchronized. A thread mutating the collection might update modCount in its local CPU cache, but that update may not be immediately flushed to main memory. Another iterating thread might read a stale value, missing the modification.
  • Non-Structural Changes: Modifying the state of an object already inside the collection (e.g. list.get(0).setName("Bob")) does not change list size, so modCount is not incremented and no CME is thrown.

5. Fail-Safe / Weakly Consistent Iterators

The term "fail-safe" is a common design pattern name (not a term used in the Javadoc; Java Javadoc refers to them as **weakly consistent**). Fail-safe iterators traverse a copy or use direct bucket pointer safety to avoid throwing CME.

CopyOnWriteArrayList (Snapshot Iteration)

When write operations occur (like add()), this collection clones the backing array, writes to the clone, and swap-references the volatile array pointer.
The iterator holds a reference to the **original array snapshot** at the time of creation. Since that snapshot is never mutated, the iterator is 100% safe from CME.
Trade-off: High write memory overhead (cloning arrays) and stale read views (the iterator will not see additions made after it was created).

ConcurrentHashMap (Bucket Traversal)

Iterators here do not copy arrays. Instead, they traverse bucket node chains directly using volatile reads. They reflect some, but not all, subsequent map updates and are guaranteed to never throw CME.

6. Key Interview Questions

Q: Can a fail-safe iterator reflect list updates after its creation?
It depends on the collection. In CopyOnWriteArrayList, **no**—it iterates over a static array snapshot. In ConcurrentHashMap, **yes**—it traverses active buckets directly using weakly consistent reads.
Q: What is the main structural trade-off between Fail-Fast and Fail-Safe?
Fail-fast iterators are memory-efficient (O(1) space) and detect bugs quickly, but crash under concurrency. Fail-safe iterators are concurrent-friendly but incur higher memory overhead and may traverse stale data.
