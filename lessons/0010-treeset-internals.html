TreeSet Internals

~11 minutes · builds on HashSet internals

The Backing TreeMap

Like HashSet, a TreeSet does not implement its core data structures from scratch. It is a wrapper delegating all elements to a backing TreeMap instance as keys, with the sentinel object PRESENT as values:

public class TreeSet<E> extends AbstractSet<E> implements NavigableSet<E> {
    private transient NavigableMap<E,Object> m;
    private static final Object PRESENT = new Object();
    
    public TreeSet() {
        this(new TreeMap<>());
    }
}

Under the hood, a TreeMap is implemented as a Red-Black Tree: a self-balancing binary search tree. Each node contains a key, a value, and links to left, right, and parent nodes, plus a boolean flag representing its color (Red or Black).

Source: Oracle TreeSet API Specifications

Self-Balancing Binary Search Trees

In a standard binary search tree, search time depends on node layout. If keys are added in sorted order (e.g. 10, 20, 30), the tree degenerates into a straight linked list, degrading search performance to linear O(n) time.

A Red-Black Tree enforces structural rules (such as maintaining black height balance and avoiding adjacent red nodes) that trigger tree rotations on insertion. This guarantees that the tree height remains bounded by approximately 2 log(n + 1). As a result, lookups, insertions, and deletions are guaranteed to scale at logarithmic O(log n) time.

Role of Ordering: Comparable vs. Comparator

Because elements must be sorted, a TreeSet requires a way to compare them. It utilizes one of two interfaces:

  • Comparable (Natural Ordering): The element's class implements java.lang.Comparable and overrides compareTo(E other). Standard classes like String, Integer, and Date implement this by default.
  • Comparator (Custom Ordering): You pass a custom java.util.Comparator implementation to the TreeSet constructor, defining comparison rules externally (e.g. comparing Strings by length instead of alphabetical order).
Gotcha: Comparison vs. equals()

A HashSet checks if an item is a duplicate using equals(). A TreeSet ignores equals() entirely during insertion; duplicate check is strictly based on the comparison result. If compareTo(x) == 0 (or compare(a, b) == 0), the element is rejected as a duplicate, even if equals() returned false!

Retrieval Practice

Cover the answer. If you need a collection that stores unique values and you only need lookup, insertion, and deletion operations, which collection should you select: HashSet or TreeSet?

Reveal

You should select HashSet. HashSet operates in average constant O(1) time for lookup, insertion, and deletion, which is faster than the logarithmic O(log n) performance of TreeSet. You should only choose TreeSet if you specifically need elements to be sorted or require range operations (like extracting sub-sets).
