Resize and Load Factor

~8 minutes · builds on buckets & put()

Why resize exists

As entries accumulate, buckets get crowded — collision chains grow and get/put slow down. HashMap grows the table array to spread entries across more buckets.

Source: Oracle HashMap — Implementation Notes

Load factor

Load factor How full the table is allowed to get before doubling. Default 0.75f — a time/space tradeoff chosen by the JDK authors. 
threshold = capacity × loadFactor

// defaults: capacity 16, load factor 0.75
threshold = 16 × 0.75 = 12

Higher load factor → fewer empty buckets, less memory, longer chains. Lower load factor → more buckets, more memory, shorter chains.

Baeldung — HashMap Load Factor

When resize fires

After every successful put, HashMap checks:

if (++size > threshold)
    resize();   // double capacity, rehash every entry

With defaults: entries 1–12 fit without resize. The 13th insert makes size = 13 > 12 → resize to capacity 32, new threshold 24.

What resize actually does

  1. Allocate new array with double the capacity
  2. For each entry in the old table, recompute index — mask changed from 15 to 31
  3. Move entries into new buckets (chains may split across two indices)
Before resize (capacity 4, mask 011): index 2: A → C → null After resize (capacity 8, mask 1111): index 2: A → null (stayed — extra bit was 0) index 6: C → null (moved — extra bit was 1)

Each entry either stays or moves to oldIndex + oldCapacity. Cached hash in each Node avoids recomputing hashCode() during this walk.

Cost of resize

Resize is O(n) — every entry is revisited. Frequent resizes on a growing map cause latency spikes. If you know the final size upfront, set initial capacity to avoid them:

// Java 19+: sized for ~100 entries without resize
Map<String, Integer> map = new HashMap<>(100);

// Older pattern: capacity = expectedSize / 0.75 + 1
int cap = (int) (expectedSize / 0.75f) + 1;
new HashMap<>(cap);

Oracle: "If the initial capacity is greater than the maximum number of entries divided by the load factor, no rehash operations will ever occur."
