HashMap: Buckets and put()

~10 minutes · mission · first of HashMap series

Not magic — an array of buckets

A HashMap is a hash table: an array called table, each slot a bucket. Each bucket holds zero or more key-value entries.

table (capacity 8): [0] null [1] null [2] (hash|keyA|valA|next) → (hash|keyD|valD|null) ← collision chain [3] null [4] (hash|keyB|valB|null) ...

Each entry is a Node: cached hash, key, value, and next pointer for collisions.

Source: Oracle HashMap javadoc

Step 1 — spread the hash

Java doesn't use key.hashCode() raw. It spreads high bits into low bits so poor hashCode() implementations still distribute:

int h = key.hashCode();
int hash = h ^ (h >>> 16);

Step 2 — pick a bucket index

Capacity is always a power of 2 (default 16). Index uses bitwise AND — fast modulo:

int index = (capacity - 1) & hash;
// capacity 16 → index = hash & 15  →  always 0..15

Step 3 — put() at that bucket

Three outcomes when you call map.put(key, value):

CaseWhat happens
Bucket emptyCreate a Node, store at table[index]
Same keyhash matches and key.equals() → replace value
Different key, same indexCollision — link new node via next (tail of chain)
map.put("Alice", 1);  // bucket 5 empty → Node stored
map.put("Bob",   2);  // bucket 2 empty → Node stored
map.put("Carol", 3);  // bucket 5 again → chain: Alice → Carol

Collisions are normal — two different keys can land in the same bucket. The map walks the chain with equals to find the right one.

get() is the mirror

V get(K key) {
    int index = (capacity - 1) & spreadHash(key);
    for (Node n = table[index]; n != null; n = n.next)
        if (n.hash == hash && key.equals(n.key))
            return n.value;
    return null;
}

Check hash first (fast), then equals (correct).

Practice — what happens on put?

Bucket at index 3 already has one entry for key "X". You call put("Y", 99) and "Y".hashCode() maps to index 3, but "Y".equals("X") is false.

What does HashMap do next?

Retrieval — explain in one sentence

Cover the answer. Why does HashMap store hash inside each Node instead of calling key.hashCode() on every comparison?

Reveal

During resize, keys are re-indexed — cached hash avoids recomputing. On lookup, comparing cached hash before equals skips expensive equality checks when hashes differ.
