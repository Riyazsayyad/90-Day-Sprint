Deep Dive: equals() and hashCode() in Java

~25 minutes · ties to mission

1. Why java.lang.Object Provides equals() and hashCode()

In Java, java.lang.Object is the common ancestor of all classes. By putting equals(Object) and hashCode() directly in the root class, Java establishes a universal capability: every object can be compared for equivalence and can represent itself as an integer hash.

This allows Java's core library classes—specifically collections like HashMap, HashSet, and Hashtable—to work polymorphically with any class you ever write, without needing custom wrappers or explicit comparison adapters.

2. Reference Equality (==) vs. Logical Equivalence (.equals())

Understanding the difference between reference comparison and logical comparison is critical for bug-free code:

  • == (Reference Equality): Compares memory addresses. It evaluates to true if and only if both variables point to the exact same object instance on the heap.
  • equals() (Logical Equivalence): Intended to check if two distinct objects are "meaningfully" equal. It compares the internal state (fields) of the objects.
Example: 
String s1 = new String("Java");
String s2 = new String("Java");
System.out.println(s1 == s2);      // false (different heap instances)
System.out.println(s1.equals(s2)); // true (same character data state)

3. Default Implementations in java.lang.Object

If you do not override these methods, you inherit the default behavior from java.lang.Object:

  • Default equals(Object obj): Checks reference equality using ==. 
    public boolean equals(Object obj) {
    return (this == obj);
}
  • Default hashCode(): A public native int hashCode(). The JVM generates this value, typically by mapping the memory address of the object to an integer (often using the identity hashcode, cached in the object's Mark Word header).

4. The Crucial Contracts and Why They Matter

Java enforces strict mathematical rules on how these methods must behave. If your implementation violates these rules, collections will silently break, causing memory leaks, duplicate keys, and missing records that are impossible to find at runtime.

The equals(Object obj) Contract

For any non-null references x, y, and z, the implementation must be:

  1. Reflexive: x.equals(x) must always return true.
  2. Symmetric: If x.equals(y) is true, then y.equals(x) must also be true.
  3. Transitive: If x.equals(y) is true and y.equals(z) is true, then x.equals(z) must be true.
  4. Consistent: Multiple invocations of x.equals(y) must return the same result, provided no information used in equals comparisons is modified.
  5. Non-nullity: x.equals(null) must always return false.

The hashCode() Contract

  1. Whenever it is invoked on the same object more than once during an execution of a Java application, the hashCode method must consistently return the same integer (provided no information in equals is modified).
  2. If two objects are equal according to the equals(Object) method, then calling the hashCode method on each of the two objects must produce the same integer result.
  3. It is not required that if two objects are unequal, they must have different hashcodes. However, producing distinct integers for unequal objects improves the performance of hash tables.

5. How Hash-Based Collections Use equals() and hashCode()

Let's demystify how a HashMap retrieves or inserts values. A HashMap uses an array of buckets internally. When you call map.put(key, value) or map.get(key):

Step 1: Hashing & Indexing

The map first invokes key.hashCode() to obtain the raw hash. It then applies a supplemental hash function to prevent low-quality hashes from clustering, and performs a modulo-like bitwise operation to find the target bucket index:
index = (n - 1) & hash (where n is the bucket array length).

Step 2: Bucket Search

Once at the bucket, the map traverses the nodes (either a linked list or a red-black tree). To identify if the key already exists, it performs two comparisons on each node's key:

  1. It compares the 32-bit hash integers: node.hash == hash (fast check).
  2. If and only if the hashes match, it executes node.key == key || node.key.equals(key).

Key Takeaway: Hash comparison acts as a fast guard. If two keys have different hashcodes, they are in different buckets, so equals() is never called! If hashcodes match, equals() is the final authority to prove equivalence.

6. Broken Implementations & What Happens

Case A: Override `equals()` but NOT `hashCode()`

Suppose you have a User class and override equals() to check if usernames match, but you leave the default hashCode().
If you create two instances new User("riyaz") and new User("riyaz"), they are equal logically. But their default hashcodes will differ.
When you store the first instance in a HashSet, it goes into Bucket 3.
When you call set.contains(new User("riyaz")), the second instance generates a different hash code (say, Bucket 7). The set searches Bucket 7, finds it empty, and returns false. You now have duplicates in your set!

Case B: Override `hashCode()` but NOT `equals()`

If you override hashCode() to return 42 for all instances, but leave default equals():
All instances land in the exact same bucket, causing severe collisions.
When you call map.get(new User("riyaz")), the map finds the bucket, hashes match (both are 42), and then calls equals(). Because equals() defaults to == reference comparison, it returns false. The map fails to find your user, rendering the data irretrievable.

7. Performance Implications: Hash Collisions

A poor hashcode distribution (e.g., returning a constant or ignoring important fields) maps many keys to the same bucket.
In Java 8+, if a bucket's linked list size exceeds 8, the map converts it into a Red-Black Tree (treeification).

  • Ideal Performance: O(1) constant time for operations when keys hash uniformly.
  • Degraded Performance: O(log N) logarithmic time for treeified buckets, or O(N) linear time for untreeified linked lists.

8. The Danger of Mutable Keys

Crucial Rule: Never use mutable objects as keys in a hash-based collection, or at least never modify them after insertion.

If you insert an object into a HashMap and subsequently modify a field used in its hashCode() calculation:

  1. The object's hashcode changes.
  2. The object remains physically stored in the bucket corresponding to its old hashcode.
  3. If you attempt to retrieve the object using the modified key, the map searches the bucket corresponding to the new hashcode, misses it, and returns null.
  4. If you try to retrieve it using the old key values, the map computes the old hash, goes to the correct bucket, but then calls equals(). The fields have changed, so equals() returns false. The element is now a memory leak, stuck in the map forever!

9. Code Examples: Bad vs. Good

❌ The Buggy Implementation

This class violates symmetry, transitivity, and breaks when added to collections due to a missing/bad hashCode.

public class BadEmployee {
    private int id;
    private String name;

    public BadEmployee(int id, String name) {
        this.id = id;
        this.name = name;
    }

    // VIOLATION 1: Subclass check breaks symmetry if inheritance is used
    @Override
    public boolean equals(Object obj) {
        if (obj instanceof BadEmployee) {
            return this.id == ((BadEmployee) obj).id;
        }
        return false;
    }

    // VIOLATION 2: No hashCode overridden! 
    // Inherits default identity hashCode, breaking HashMap lookups.
}

✅ The Correct Modern Implementation (Java 7+)

import java.util.Objects;

public final class GoodEmployee {
    private final int id;
    private final String name;

    public GoodEmployee(int id, String name) {
        this.id = id;
        this.name = name;
    }

    @Override
    public boolean equals(Object obj) {
        // 1. Optimize: Check if references are identical
        if (this == obj) return true;

        // 2. Check nullity and exact class match (avoids subclass symmetry issues)
        if (obj == null || getClass() != obj.getClass()) return false;

        // 3. Safe cast
        GoodEmployee other = (GoodEmployee) obj;

        // 4. Compare fields (using null-safe Objects.equals)
        return this.id == other.id && 
               Objects.equals(this.name, other.name);
    }

    @Override
    public int hashCode() {
        // Returns a hash derived from all fields checked in equals
        return Objects.hash(id, name);
    }
}

10. Objects.equals() and Objects.hash()

Introduced in Java 7, java.util.Objects provides static helpers that simplify implementation:

  • Objects.equals(a, b): Handles null values gracefully. It returns true if both are null, false if only one is null, and delegates to a.equals(b) otherwise.
  • Objects.hash(Object... values): Accepts a varargs array and computes a hash value. Internally, it calls Arrays.hashCode() which uses the formula:
    h = 31 * h + (value == null ? 0 : value.hashCode()).

Performance Tip: While Objects.hash() is elegant, it allocates a temporary array for the varargs argument and boxes primitives. In hot loops, write an inline hash computation to avoid garbage collection overhead.

11. Java Records (Java 14+)

Java Records represent pure data carriers. When you declare a record: 

public record UserRecord(int id, String username) {}
The compiler automatically generates private final fields, accessor methods, and implicit, correct overrides of equals() and hashCode().
The generated implementations compare all components declared in the record header. If you need special rules (e.g., ignoring a field), you can manually override them within the record body, but for 95% of use cases, the compiler-generated ones are perfect and bug-free.

12. IDE-Generated vs. Lombok vs. Manual

Approach Pros Cons
Manual Handwritten Zero overhead, customized field comparison sequence. Boilerplate, prone to human error and null pointer risks.
IDE Generated (IntelliJ/Eclipse) Explicit visibility, easy to customize. Clutters source files, doesn't update automatically if fields are added.
Lombok (@EqualsAndHashCode) Clean, zero boilerplate, automatically updates. Requires compiler plugin, opaque runtime bytecode.
Objects Utility Methods Null-safe, readable, standard JDK. Slight allocation overhead in hashCode (varargs arrays).

13. Real-World Framework Integration

  • Hibernate/JPA Entities: Hibernate relies heavily on equals/hashCode. But database entities present a unique challenge: their ID is often null until persisted!
    • If you use the database ID in hashCode(), saving the entity will change its ID from null to a generated value, changing the hashcode and breaking collections (the mutable key hazard!).
    • Best Practice: Use a stable, unique business key (like UUID or natural unique fields) for equals and hashCode. If none exists, fall back to a constant hashCode (e.g. 31) and compare equality by ID if present, otherwise by reference. This forces Hibernate to use reference matching for unpersisted entities but preserves collection lookups.
  • Spring Caching (@Cacheable): When caching method results, Spring generates cache keys based on method arguments. If custom classes are arguments, they must implement a correct equals() and hashCode(), otherwise Spring cannot match the cache keys, leading to cache misses.

Common Interview Questions

Q1: Can two unequal objects return the same hashCode?

Yes. This is called a **hash collision**. Hash functions map infinite object combinations to a finite 32-bit integer space. While valid, collisions degrade hash table performance from O(1) towards O(log N) or O(N).

Q2: Why does equals() take Object as an argument rather than the specific class type?

Because the method signature in the parent class java.lang.Object is equals(Object). If you write public boolean equals(MyClass other), you are **overloading** the method, not **overriding** it. The collections library calls equals(Object), so your custom overloaded version will be ignored!

Q3: How does subclassing affect equals() transitivity?

As Joshua Bloch details in Effective Java, there is no way to extend an instantiable class and add a value component while preserving the equals contract. If you use instanceof, subclasses compare equal to parents but not vice-versa (symmetry broken), or transitivity breaks. To solve this, favor composition over inheritance, or check types using getClass() != obj.getClass() to enforce strict type identity.
