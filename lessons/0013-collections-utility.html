The Collections Utility Class

~7 minutes · ties to mission

Collection vs. Collections

In Java, it is easy to confuse the two names:

  • java.util.Collection is the **root interface** of the collections hierarchy (implemented by List, Set, Queue).
  • java.util.Collections is a **utility class** consisting exclusively of static methods that operate on or return collections. It cannot be instantiated.

1. Sorting & Searching

Two of the most frequently used methods are for ordering and retrieving elements:

  • Collections.sort(List<T> list): Sorts elements using the objects' natural order (Comparable) or a custom Comparator.
    Stable Sorting: The sort algorithm is **stable** (Timsort in modern JDKs). Equal elements are guaranteed to remain in their original relative positions after sorting.
  • Collections.binarySearch(List<?> list, T key): Performs an optimized logarithmic search.
    Prerequisite: The list **MUST be sorted** in ascending order before invoking this method. If it is not sorted, the returned index is undefined and unpredictable.
    Note on Lists: Runs in O(log n) on RandomAccess lists (ArrayList). However, on sequential-access chains (LinkedList), it traverses nodes step-by-step, degrading to O(n).

2. General Utility Methods

The class contains a wide variety of static helper algorithms for manipulating list and collection states:

Method Purpose & Key Behaviors
reverse(list) Reverses the order of elements in the specified list.
shuffle(list) Randomly permutes elements using a default or custom source of randomness.
swap(list, i, j) Swaps the elements at index i and index j in the list.
fill(list, obj) Replaces all elements in the list with the specified element obj.
copy(dest, src) Copies all elements from src to dest. **Dest must be at least as long as src**, or IndexOutOfBoundsException is thrown.
min(coll) / max(coll) Returns the minimum or maximum element according to natural order or custom comparator.
frequency(coll, obj) Returns the number of elements in the collection equal to the specified object.
disjoint(c1, c2) Returns true if the two collections have no elements in common.

3. Wrappers: List Control View Gates

Three key wrappers intercept calls to control a collection's properties, returning a thin delegating view:

  • unmodifiableList(list): Intercepts write calls (e.g., add(), remove()) and throws UnsupportedOperationException.
  • synchronizedList(list): Wraps all calls inside a synchronized(mutex) block. *Note:* manual synchronization on the mutex is required when using an iterator loop.
  • checkedList(list, Class): Validates elements added to the list at runtime, preventing generic type evasion and heap pollution.

4. Wrapper View vs. True Immutable Collection

It is vital to understand the difference between wrapper views and modern immutable collections (like List.of()):

List<String> backing = new ArrayList<>();
backing.add("Apple");

// 1. Wrapper View (Unmodifiable)
List<String> wrapper = Collections.unmodifiableList(backing);

// 2. Immutable Collection
List<String> immutable = List.copyOf(backing);

// Modifying the backing list...
backing.add("Banana");

System.out.println(wrapper);   // prints [Apple, Banana] (reflects changes!)
System.out.println(immutable); // prints [Apple]         (truly immutable!)
  • Wrapper View: Reads directly from the underlying mutable backing collection. It is a write-gate, not a data copy.
  • Immutable Collection: Independent data copy. It doesn't rely on a backing collection, rejects null values entirely, and is heavily optimized for space.
