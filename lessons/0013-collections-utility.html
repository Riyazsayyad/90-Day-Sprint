The Collections Utility Class

~6 minutes · ties to mission

The Core Difference: Collection vs. Collections

In Java, it is easy to confuse the two names:

  • java.util.Collection is the **root interface** of the collections hierarchy (implemented by List, Set, Queue).
  • java.util.Collections is a **utility class** consisting exclusively of static methods that operate on or return collections. It cannot be instantiated.

1. Wrapper Collections (Views, Not Copies)

One of the most important concepts of the Collections utility is its wrapper methods (e.g., unmodifiableList(list)). Under the hood, these wrappers are views. They do not copy the collection's data structure—they wrap the existing collection, delegating reads to it and blocking modifications.

List<String> backing = new ArrayList<>();
backing.add("Java");

List<String> view = Collections.unmodifiableList(backing);
System.out.println(view); // prints [Java]

// Modifying the backing list affects the view!
backing.add("Kotlin");
System.out.println(view); // prints [Java, Kotlin]

view.add("Go"); // Throws UnsupportedOperationException!

Gotcha: If you want true immutability (where the content cannot change at all), copy the data using modern factory methods like List.copyOf(backing) or List.of() instead.

2. Synchronized Wrappers & The Iterator Loophole

Synchronized wrappers (e.g., synchronizedList(list)) wrap all reading and writing methods in a synchronized(mutex) block, making individual operations thread-safe.

However, iteration is not thread-safe by default because iterating is a compound operation. You must manually synchronize on the wrapper's mutex when iterating to prevent race conditions or ConcurrentModificationException:

List<String> syncList = Collections.synchronizedList(new ArrayList<>());

// Thread-safe iteration pattern
synchronized (syncList) {
    for (String item : syncList) {
        System.out.println(item);
    }
}

3. Checked Wrappers

Checked wrappers (e.g., checkedList(list, String.class)) enforce type safety at runtime. They verify the class of any element being added, throwing a ClassCastException if an invalid type is passed. This is useful for catching bugs when passing generic collections to legacy, non-generic libraries.

4. Core Static Algorithms

Collections implements standard algorithms optimized for collection interfaces:

  • Collections.sort(list): Sorts elements using a hybrid stable merge sort (Timsort). Runs in O(n log n).
  • Collections.binarySearch(list, key): Performs a binary search.
    • If the list implements RandomAccess (like ArrayList), it runs in O(log n).
    • If it is sequential-access (like LinkedList), it traverses nodes step-by-step, degrading performance to O(n).
