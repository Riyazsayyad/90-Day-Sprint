Java Memory Model: Stack vs. Heap

~10 minutes · ties to mission

1. Why Java Separates Stack and Heap Memory

The JVM separates memory into distinct zones to optimize performance, manage resource lifetimes, and support thread safety:

  • Stack: Optimized for speed and immediate execution context. It runs on a LIFO (Last-In-First-Out) model bound directly to executing threads. Deallocation is automatic and instant when a method returns, avoiding fragmentation.
  • Heap: Optimized for sharing and dynamic allocation lifetimes. Since objects need to survive beyond the methods that create them, they are stored on the Heap, accessible by all threads. Reclamation is managed asynchronously by the Garbage Collector.

2. What Is Stored in the Stack?

Stack memory is allocated per thread. It stores:

  • Primitive local variables: Actual values of primitive types (e.g., int x = 5;) declared inside methods.
  • Reference variables: Memory address references (e.g. myObj) pointing to objects located on the heap.
  • Execution Stack Frames: Metadata representing nested method calls.

3. What Is Stored in the Heap?

Heap memory is shared across all threads. It stores:

  • All Objects: Concrete instances of classes, regardless of where they are created. This includes all arrays (since arrays are treated as objects in Java).
  • Instance variables: Member variables declared within a class (e.g. private int age;). Even if an instance variable is a primitive, it sits inside the object shell on the Heap.

4. Method Call Stack & Stack Frames

Every time a thread invokes a method, the JVM pushes a new **Stack Frame** onto that thread's call stack. A frame is a temporary storage block containing:

  • Local Variable Array: Indexed slots containing method parameters and local variables.
  • Operand Stack: A workspace used to perform intermediate evaluation steps (e.g., adding numbers).
  • Frame Data: Reference to the runtime constant pool, return method instructions, and exception dispatch handlers.

When a method finishes, its stack frame is popped off and discarded, instantly reclaiming the local variable slots without GC involvement.

5. Local Variables vs. Instance Variables

Feature Local Variables Instance Variables
Declaration Inside a method scope. Inside class scope (outside methods).
Location Stack frame. Heap (inside parent object).
Defaults None. Must be initialized before use. Initialized to default values (0, false, null).

6. Trace Code Execution & Memory

Analyze how the JVM maps execution step-by-step:

void run() {
    int age = 25;
    User user = new User("Alice");
}

At execution runtime:

  1. The JVM pushes run() frame to the Stack.
  2. age (primitive 25) is written directly into local slot 1 on the Stack.
  3. A new User object is allocated on the Heap, carrying its instance variables (like name).
  4. The reference variable user is created in stack slot 2, storing the heap memory address (e.g. 0x7f9a).

7. Object Lifetime & GC Eligibility

An object survives on the Heap as long as it is reachable from active roots (such as local variables in active stack frames, static variables, or JNI references).
An object becomes **eligible for Garbage Collection** the moment it has zero strong references pointing to it, or it becomes isolated in an unreachable reference group (island of isolation).

User user = new User("Bob"); // Reachable
user = null; // Original Bob object is now eligible for GC

8. StackOverflowError vs. OutOfMemoryError

  • StackOverflowError (SOE): Occurs when the Stack runs out of space, typically due to deep or infinite recursion. Each nested call pushes a frame; if calls never return, the stack boundary is breached. Controlled by JVM flag -Xss.
  • OutOfMemoryError (OOM): Occurs when the Heap space is exhausted. This happens if you instantiate too many objects that remain referenced (a memory leak), preventing the GC from reclaiming them. Controlled by JVM flag -Xmx.

9. Escape Analysis Optimization

In modern JVMs, the JIT (Just-In-Time) compiler analyzes object allocations to optimize performance.
If an object is created within a method and **never escapes** its scope (meaning it is not returned, passed to other threads, or stored in instance fields):

  • The JIT compiler can optimize this via **Scalar Replacement**, allocating individual instance fields directly on the Stack rather than creating the actual object shell on the Heap.
  • This bypasses heap allocation entirely, removing Garbage Collection pressure and speeding up execution.

10. Key Interview Questions

Q: Does Java pass variables by value or by reference? Java is strictly pass-by-value. For primitives, it passes a copy of the actual value. For objects, it passes a copy of the reference address. The reference variable itself resides on the Stack, and its address value is cloned when passed.
Q: Where do static variables reside? Static variables belong to the class template, not to any single instance. They are stored inside the java.lang.Class instance allocated in the Metaspace (under modern JVMs).
