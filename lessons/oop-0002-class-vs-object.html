Class vs. Object: Runtime and Memory Mechanics

~10 minutes · ties to mission

1. Why Java Needs Classes

Java is a statically-typed language. The compiler must verify type safety, calculate memory offsets, and check method signatures before your code runs. A Class serves as the compile-time schema that guarantees this structure:

  • Structural Contract: It defines what variables (state) and methods (behavior) will exist in any future instance.
  • Type Verification: It allows the compiler to enforce rules, preventing errors such as calling a method that does not exist.
  • Metadata Generation: It compiles into a .class file, providing bytecode instructions and class structural metadata that the JVM reads at runtime.

2. What a Class Really Represents

At compile time, a class is a source-code template.
At runtime, a class is represented as a metadata structure stored in the JVM's Metaspace. The JVM instantiates a single instance of java.lang.Class for each loaded class. This class object holds:

  • The blueprint layout (offsets of instance fields, types of variables).
  • The **vtable (virtual method table)**, which lists addresses of executable method bytecodes.
  • Static variables (class-level state shared by all instances).

3. What an Object Really Is in Memory

An Object is a concrete entity that resides on the Heap. Physically, it is a contiguous chunk of memory structured as follows:

  • Object Header: Consists of:
    • Mark Word: Holds identity hashcode, GC age metadata, and thread synchronization lock information.
    • Klass Word: A pointer to the class metadata stored in the Metaspace, allowing the JVM to identify the object's type at runtime.
  • Instance Data: The actual values of the instance fields defined by the class.
  • Padding: Extra empty bytes added to ensure the object's size is a multiple of 8 bytes, aligning with CPU cache lines.

4. Why Objects Are Created Using new

The new keyword is a dynamic heap-allocation command. When executed, the JVM performs these steps:

  1. Calculates the memory size required for the object based on class metadata.
  2. Allocates a contiguous block of memory on the Heap.
  3. Fills the Object Header (linking the Klass Word to the Metaspace class metadata).
  4. Initializes instance variables to their default values (e.g., 0, false, null).
  5. Invokes the constructor to populate custom fields and returns the memory address (reference pointer).

5. Class Loading vs. Object Instantiation

It is vital to distinguish between loading class schemas and instantiating row objects:

  • Class Loading (Once): The JVM loads class bytecode into Metaspace via class loaders, executing static initialization blocks. This happens the first time a class is referenced.
  • Object Instantiation (Many): Every invocation of new allocates fresh heap space, sets default instance values, runs instance initializers, and runs the constructor.

6. Reference Variables vs. Actual Objects

In Java, variables never hold objects directly. They hold **references** (pointers to memory addresses):

Car myCar = new Car();
  • myCar: The reference variable, usually allocated on the Stack. It stores only a 32-bit or 64-bit address.
  • new Car(): The actual object, allocated on the Heap, housing the car's state data.

Multiple reference variables can point to the exact same object on the heap, similar to how two remote controls can operate a single television. Changes made via one reference variable are visible through the other.

7. How Multiple Objects Share the Same Class

If you instantiate 1,000 objects of the class User, the JVM does not replicate the bytecode of the methods 1,000 times.
Instead, all 1,000 objects have a header pointing to the single `User` class definition in Metaspace. When a method is called: 

userObj.login();

The JVM uses the object's header to locate the class vtable in Metaspace, resolves the method address, and executes the bytecode, passing the address of userObj as an implicit first parameter (representing this).

8. High-Level Analogies

Instead of just using the generic "blueprint" analogy, consider these models:

  • Database Schema vs. Database Row: The class is the table schema (defining columns, constraints, data types), which occupies no table space. The object is a row of data inserted into that table.
  • Recipe vs. Baked Cake: The class is a recipe (instructions, list of inputs). You cannot eat the recipe. The object is the cake baked on the counter. Multiple cakes share the same recipe but exist as distinct physical objects.

9. Key Interview Questions

Q: Does a Java class occupy memory at runtime? Yes. While it does not consume heap space for instance variables, the JVM loads class metadata (bytecode, static fields, and vtable structures) into the Metaspace, and instantiates a java.lang.Class metadata object.
Q: What is the size of an empty object on the Heap? In a standard 64-bit JVM, an empty object occupies 16 bytes: 8 bytes for the Mark Word (locking/GC), 4 or 8 bytes for the Klass Word (class pointer), and padding to align to the nearest 8-byte boundary.
