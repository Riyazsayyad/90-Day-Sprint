ArrayList Internals

~8 minutes · builds on resize & load factor

The Backing Array

Unlike a `LinkedList`, which represents elements as node chain links, Java's ArrayList stores elements inside a simple, contiguous backing array: Object[] elementData.

Because primitive Java arrays have a fixed size, the ArrayList class manages the complexity of expanding this array dynamically behind the scenes as you insert elements.

Source: Oracle ArrayList API Specifications

Capacity vs Size

To understand the mechanics, we must separate two fundamental properties of an ArrayList:

  • Capacity: The physical length of the backing array (elementData.length). This is the maximum number of items the list can hold before allocating a new array.
  • Size: The logical count of elements currently stored in the list by the user (accessed via size()).

The condition size <= capacity is always true. When size == capacity, the backing array is full, and the next insertion will trigger a growth phase.

Growth Strategy (1.5x Scaling)

When you attempt to add an element to a full list, Java reallocates a larger array. The JDK calculates the new capacity using a bitwise shift right operation:

int newCapacity = oldCapacity + (oldCapacity >> 1);

Since shifting right by 1 bit (>> 1) is equivalent to integer division by 2, this means:
newCapacity = oldCapacity + 0.5 * oldCapacity = 1.5 * oldCapacity.

Once the new capacity is computed, Java allocates the new array and copies the old elements over via Arrays.copyOf(), which internally utilizes the extremely fast, native memory copying operation System.arraycopy().

Why Random Access is O(1)

In computer memory, an array is stored as a contiguous block of bytes. Because every object reference in Java takes up the same fixed amount of space (typically 4 or 8 bytes), the JVM can compute the exact physical address of any element immediately without traversing the list:

Address(index) = BaseAddress + index * ReferenceSize

This single addition and multiplication allows the computer to look up the element at any arbitrary index in constant O(1) time, whether it is the first element, the middle element, or the last.

Insertions and Deletions

While indexing is fast, altering the size of the array has trade-offs:

  • Inserting/Deleting at the end: This takes O(1) time. Resizing occurs only occasionally (at a geometric interval), so the cost of copies amortizes to a constant O(1) overall.
  • Inserting/Deleting at arbitrary indices: This takes O(n) time. If you insert at index i, all elements from index i to size - 1 must be shifted one slot to the right. Deleting an element requires shifting everything to the left. Try this in the simulator above and watch the orange highlight shift step-by-step.

Retrieval Practice

Cover the answer. What is the time complexity of looking up a value in an ArrayList if you do not know its index (e.g. searching for a value using contains(Object o))?

Reveal

It is O(n). Because you do not know the index, the list must iterate through the backing array one-by-one from index 0 until the matching element is found.
