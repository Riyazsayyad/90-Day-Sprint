Inheritance: Hierarchy, Chaining, & Method Dispatch

~12 minutes · ties to mission

1. Why Inheritance Exists & The IS-A Relationship

In object-oriented system design, **Inheritance** is the mechanism by which one class acquires the state (fields) and behavior (methods) of another class. It serves two primary architectural goals:

  • Code Reuse: Subclasses inherit verified, tested code from parent classes, eliminating duplicate logic.
  • Polymorphic Typing (The IS-A Contract): By extending a class, you establish a semantic contract: a subclass instance *is-a* type of the superclass. A StripeGateway *is-a* PaymentGateway.

According to the Liskov Substitution Principle (LSP), a subclass must be completely substitutable for its superclass without breaking client expectations. If extending a class violates the IS-A contract, prefer **Composition** (wrapping a reference to another class) instead.

2. Constructor Chaining under the Hood

Subclasses do not inherit constructors (which are specific to initializing their own declaring class). However, subclass constructors must chain to a parent constructor to guarantee that the inherited parent state is fully initialized first. Java enforces these rules:

  • First Line Restriction: The first statement of a subclass constructor must be an explicit call to another constructor in the same class (using this()) or to the superclass constructor (using super()).
  • Compiler Default insertion: If you do not write an explicit call, the compiler automatically inserts a no-argument super() call as the first line. If the parent class defines a parameterized constructor and lacks a default no-argument constructor, a compilation error occurs unless you write super(...) explicitly.

3. Method Overriding Rules

Method Overriding allows a subclass to provide a specific implementation for a method already defined in its superclass. To override a method, the subclass version must adhere to these strict contracts:

  1. Same Signature: Must have the exact same name and parameter list. Use the @Override annotation to force the compiler to validate this.
  2. Covariant Return Type: The return type can be a subclass (subtype) of the type declared in the parent method.
  3. Access Modifiers: Access cannot be more restrictive (e.g. you cannot override a protected method and make it private).
  4. Exception List: The overridden method cannot throw new or broader checked exceptions, though it can throw fewer, narrower, or unchecked exceptions.

Note: Methods declared as private, final, or static cannot be overridden.

4. Dynamic Method Dispatch & The vtable

Also known as Runtime Polymorphism, **Dynamic Method Dispatch** is the mechanism by which a call to an overridden method is resolved at runtime rather than compile time.
The reference variable type determines which methods are accessible at compile time, but the actual object type on the heap determines which overridden implementation executes at runtime:

PaymentGateway gateway = new StripeGateway(); // Reference = Parent; Heap = Child
gateway.processPayment(100.0); // Executes StripeGateway's overridden method

Under the hood, the JVM executes this lookup using a vtable (Virtual Method Table). Every class loaded into Metaspace has a vtable listing pointers to its virtual methods. If a subclass overrides a parent method, the JVM replaces the parent's method pointer with the subclass method pointer in the subclass vtable, enabling O(1) runtime method lookup.

5. Single Inheritance & The Diamond Problem

Java classes support **Single Inheritance** (a class can extend at most one class) to prevent ambiguity. In languages with multiple inheritance, the **Diamond Problem** occurs when class A defines a method, classes B and C override it, and class D extends both B and C. If D calls the method, which parent implementation should it run?

To avoid this conflict, Java forbids class-level multiple inheritance. However, Java allows multiple inheritance of interfaces. If default method implementations conflict across interfaces, the compiler forces the subclass to override the conflicting method and resolve it explicitly:

public class Device implements Camera, Scanner {
    @Override
    public void start() {
        Camera.super.start(); // Explicit resolution of default method conflict
    }
}

6. Why Constructors Cannot Be Inherited or Overridden

  • No Inheritance: Constructors initialize the specific fields of the class they are declared in. If constructors were inherited, a subclass could run a parent constructor that knows nothing about the subclass's new fields, leaving them half-initialized.
  • No Overriding: Overriding requires matching method signatures (which includes matching the name). A constructor must be named exactly after its declaring class. Because a subclass and superclass have different names, a subclass constructor can never have the same name as a superclass constructor, making overriding syntactically impossible.

7. Real-World Example: UI Component Rendering

Here is an enterprise class hierarchy demonstrating method overriding, constructor chaining, and dynamic dispatch:

public abstract class UIComponent {
    private final String id;
    private boolean visible = true;

    protected UIComponent(String id) {
        this.id = id; // Parent state initialized
    }

    public void render() {
        if (visible) {
            drawBorder();
            drawContent();
        }
    }

    protected void drawBorder() {
        System.out.println("Drawing generic component border");
    }

    protected abstract void drawContent(); // Child must implement
}

public class UIButton extends UIComponent {
    private final String label;

    public UIButton(String id, String label) {
        super(id); // Explicit constructor chaining
        this.label = label; // Child state initialized
    }

    @Override
    protected void drawContent() {
        System.out.println("Rendering Button Label: " + label);
    }

    @Override
    protected void drawBorder() {
        System.out.println("Drawing custom rounded button border"); // Override
    }
}

8. Key Interview Questions

Q: Can you override static methods in Java? No. If a subclass defines a static method with the same signature as a superclass static method, it is called Method Hiding, not overriding. The method called is determined statically at compile time based on the reference type, not the heap object type.
Q: What happens if a superclass constructor calls an overridable method? If a parent constructor calls an overridden method, the subclass overridden implementation runs *before* the subclass constructor has initialized its own fields. If the overridden method accesses subclass fields, they will return default null/zero values, leading to severe bugs or NullPointerException.
