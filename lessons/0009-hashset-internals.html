HashSet Internals

~10 minutes · builds on HashMap internals

The Backing HashMap

In Java, a HashSet does not implement hashing logic or collision buckets from scratch. Instead, it is a simple wrapper around a HashMap. When you instantiate a HashSet, it secretly allocates an internal HashMap instance:

public HashSet() {
    map = new HashMap<>();
}

Every element you add to a HashSet is stored as a key in this backing map. Because a map requires both keys and values, the set uses a single static sentinel dummy object called PRESENT as the value for every entry:

private static final Object PRESENT = new Object();

Source: Oracle HashSet Specifications

How Duplicates are Rejected

A key property of a Set is that it cannot contain duplicate elements. This rule is naturally enforced by the backing map's key uniqueness guarantee.

When you call hashSet.add(element), it delegates directly to map.put(element, PRESENT):

public boolean add(E e) {
    return map.put(e, PRESENT) == null;
}

Recall the behavior of HashMap.put(K key, V value):

  • If the key is new: The map stores it, returns null. The set checks null == null and returns true (item added).
  • If the key is a duplicate: The map replaces the old value with PRESENT (which changes nothing) and returns the previous value (which is PRESENT). The set checks PRESENT == null, which evaluates to false (item rejected).

Role of equals() and hashCode()

To identify if a key is a duplicate, the backing HashMap needs a way to locate and compare objects. It does this in two steps using the element's hashCode() and equals() methods:

  1. Locate the bucket: It hashes the element and finds the bucket slot index.
  2. Compare key references: If the bucket already contains nodes, the map walks the chain, checking if (node.hash == hash && (node.key == key || key.equals(node.key))).

Because of this, if you store custom objects in a HashSet, you must override both hashCode() and equals(). If you override only one, or fail to override them consistently, the set will either allow duplicate entries or fail to locate elements during lookup.

Time Complexities

Because all operations delegate directly to the backing HashMap, they share its performance characteristics:

  • add(), remove(), contains():
    • Average case: O(1). If hash codes are well-distributed, elements map to distinct buckets with short chains, requiring constant-time lookups.
    • Worst case: O(log n) (Java 8+) / O(n) (Java 7-). Under severe hash collisions where all elements hash to the same bucket index, the chain grows linear in size. In Java 8+, the bucket treeifies into a Red-Black Tree, capping search time to logarithmic complexity.

Retrieval Practice

Cover the answer. Why is the dummy value PRESENT defined as a static member (private static final Object PRESENT = new Object();) instead of creating a new Object instance inside each HashSet or for each element?

Reveal

To conserve memory. Since the backing HashMap requires a value for every node, creating a new dummy object instance per element would waste heaps of memory. Defining PRESENT as a static member creates exactly one instance of Object in heap memory, shared across all keys and all instances of HashSet in the application.
