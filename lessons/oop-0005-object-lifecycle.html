Object Lifecycle in Java: Allocation to Reclamation

~10 minutes · ties to mission

1. The Five Stages of the Object Lifecycle

Every Java object passes through five distinct phases in JVM memory:

  1. Class Loading: The JVM loads, links, and initializes class bytecode in Metaspace (executing static members). This must occur once before instantiation can proceed.
  2. Instantiation & Allocation: The new bytecode allocates memory on the Heap (often utilizing thread-safe Thread Local Allocation Buffers) and runs initializers and constructors to build the object.
  3. Active Usage: The object is referenced by active variables and accessed by threads to execute business logic.
  4. GC Eligibility: The object loses all strong reachability paths from GC Roots, rendering it dead in the eyes of the memory collector.
  5. GC Reclamation: The Garbage Collector identifies the unreachable object, runs finalize (if overriding and not yet run), sweeps its heap space, and returns the memory address to the free list.

2. Object Instantiation Step-by-Step

When you invoke new Car(), the JVM executes a strict sequence under the hood:

  • Metaspace Check: Verifies if the class metadata is loaded. If not, triggers class loading.
  • Heap Sizing & Allocation: Computes the required object size (header + fields + padding) and allocates heap space.
  • Memory Zeroing: Fills all instance fields with their default values (e.g. 0, false, null) to ensure structural safety.
  • Header Setup: Writes the Mark Word (locking/GC age) and Klass Word (pointer to Metaspace class schema).
  • Custom Initialization: Runs direct field assignments, instance initialization blocks, and the constructor body.

    • 3. Reachability & GC Roots

    The Garbage Collector determines if an object is alive by tracing reachability.
    The trace starts from **GC Roots** (references that are directly accessible outside the heap, such as local variables in active thread stacks, static class variables, and JNI handles).
    If there is no path of strong references connecting active GC Roots to an object, that object is declared unreachable and becomes eligible for GC. 

    User user = new User("Alice"); // Root references Alice
user = null; // Reference broken; Alice is eligible for GC

    4. finalize() Deprecation & Security Hazards

    Historically, the finalize() method was overridden to clean up resources (like closing files or database connections). However, it is deprecated (since Java 9) and removed in recent JDKs. Here is why finalize is considered a major design hazard:

    • No Timing Guarantees: The JVM does not guarantee *when* or even *if* the finalizer will execute. Resources can leak indefinitely under low GC activity.
    • Performance Degradation: Objects overriding finalize() are added to a JVM finalization queue. This forces the GC to keep them alive for at least one extra collection cycle, causing heap pressure.
    • Single-Thread Bottleneck: Finalizers execute on a single, low-priority JVM system thread. If one finalize method hangs or blocks, the entire queue backs up, leading to OutOfMemoryError.
    • Finalizer Attacks (Security): A malicious subclass can override finalize(). If instantiation fails in the parent constructor, the half-created object is still subject to finalization. The overridden finalizer can save a reference (this) to static variables, resurrecting a compromised object.

    Alternative: Implement AutoCloseable and clean up resources using try-with-resources statements, or utilize java.lang.ref.Cleaner.

    5. Object Resurrection

    If an object overrides finalize(), it has one opportunity to save itself from reclamation. In the finalizer body, if it assigns this to an active reference, it becomes reachable again:

    @Override
protected void finalize() {
    activeInstances.add(this); // Resurrected!
}

    Key Caveat: The JVM runs finalize() **at most once** per object. If the resurrected object later becomes unreachable again, the finalizer is not re-run; the object is swept immediately.

    6. High-Level Reference Strengths

    To control GC behavior, Java provides four reference strengths via package java.lang.ref:

    1. Strong Reference: Standard variable references (e.g. Object obj = new Object();). The GC never collects strongly reachable objects.
    2. Soft Reference: (`SoftReference`). Collected only when the JVM runs out of heap memory and is about to throw an OOM. Ideal for memory-sensitive caches.
    3. Weak Reference: (`WeakReference`). Collected during the very next GC run, regardless of memory capacity. Used in metadata maps (like WeakHashMap).
    4. Phantom Reference: (`PhantomReference`). Its get() method always returns null. Combined with ReferenceQueue to schedule post-mortem resource cleanup after the object is completely freed from memory.

    7. Key Interview Questions

    Q: What is a finalizer attack, and how do you prevent it? It occurs when a subclass overrides finalize() to resurrect a partially initialized object whose constructor failed. To prevent this, declare your class as final, or write a final, empty finalize() method in your class to block overrides.
    Q: How does a WeakHashMap prevent memory leaks? In a WeakHashMap, the keys are stored as weak references. If a key object is no longer referenced elsewhere in the application, the GC reclaims it, and the map automatically cleans up the corresponding entry.
