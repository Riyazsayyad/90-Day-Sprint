Collections Utility Class Reference

Oracle java.util.Collections Javadoc

1. Sorting & Searching Algorithms

Method Behavior & Internal Constraints
sort(List<T> list)
sort(List<T> list, Comparator<? super T> c)		 Sorts the list. Uses the objects' natural ordering (Comparable) or a custom ordering (Comparator).
Stable Sorting: It uses Timsort (stable hybrid merge-insertion sort), meaning equal elements retain their original relative order. Time complexity is guaranteed O(n log n).
binarySearch(List<? extends Comparable> list, T key)
binarySearch(List<T> list, T key, Comparator<? super T> c)		 Searches the list using binary search. Returns index if found; otherwise, returns -(insertion_point - 1).
Prerequisite: The list **MUST be sorted** in ascending order before searching (via the same ordering contract). If not, search results are undefined.
Performance: O(log n) on RandomAccess lists (ArrayList). Degrades to O(n) on sequential-access chains (LinkedList) due to pointer traversals.

2. General Utility Methods

  • reverse(List<?> list): Reverses the order of elements in the specified list. Runs in O(n).
  • shuffle(List<?> list): Randomly permutes elements in-place using a Fisher-Yates loop. Runs in O(n).
  • swap(List<?> list, int i, int j): Swaps the elements at indices i and j in O(1) for random access lists.
  • fill(List<? super T> list, T obj): Replaces every element in the list with the specified object. Runs in O(n).
  • copy(List<? super T> dest, List<? extends T> src): Copies elements from src to dest.
    Constraint: The destination list (dest) must be at least as long as the source list (src), otherwise it throws an IndexOutOfBoundsException.
  • min(Collection<? extends T> coll) / max(Collection<? extends T> coll): Returns the minimum / maximum element according to natural order or custom comparator in O(n) time.
  • frequency(Collection<?> coll, Object o): Returns the count of elements in the collection equal to the specified object in O(n).
  • disjoint(Collection<?> c1, Collection<?> c2): Returns true if the two collections have no elements in common. Optimized to iterate over the smaller collection and query the larger.

3. Wrapper Collections (Views, Not Copies)

The Collections wrappers return thin views representing the backing mutable collection. Writes are intercepted and blocked or synchronized, but elements are not copied.

  • unmodifiableList(list) / unmodifiableSet(set) / unmodifiableMap(map): Blocks write methods by throwing UnsupportedOperationException. However, mutations to the underlying backing collection will read through and change the view.
  • synchronizedList(list) / synchronizedSet(set) / synchronizedMap(map): Wraps all calls inside a synchronized(mutex) block. Iteration must be manually synchronized on the wrapper's mutex.
  • checkedList(list, Class) / checkedSet(set, Class): Enforces runtime type checks during item inserts, throwing ClassCastException to prevent generics heap pollution.

4. Wrapper View vs. True Immutable Collection

Feature Wrapper View (e.g., Collections.unmodifiableList) Immutable Collection (e.g., List.of, List.copyOf)
Underlying Changes Reflected (reads straight through to backing mutable collection). No backing collection exists (data is copied and locked on creation).
Null Values Allowed if the backing collection allows them. Forbidden (throws NullPointerException on creation/contains query).
Memory Overhead Slightly higher (requires two object instances: wrapper + backing list). Highly optimized, space-efficient internal class representations.
