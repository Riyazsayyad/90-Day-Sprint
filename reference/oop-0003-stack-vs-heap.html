Java Stack vs. Heap Memory Reference

Study Reference for JVM Runtime Allocations & Execution Boundaries

1. Quick Paradigm Contrast

Feature Stack Memory Heap Memory
Access Scope Private. Thread-confined (each thread has its own Stack). Global. Shared across all threads.
Allocation Style LIFO structured. Pushed/popped automatically by execution. Dynamic. Allocated via new anywhere on the heap.
Lifetime Automatic. Reclaimed when method frame pops. Reclaimed asynchronously by Garbage Collection.
Speed Extremely fast. Directly indexed via CPU instructions. Slower. Subject to pointer lookup and GC overhead.

2. Inside a Stack Frame

For every method call, the JVM creates a Stack Frame containing:

  • Local Variable Array: Indexed slots containing parameters (slot 0 contains this for instance methods) and local variables. Primitives store values; references store heap addresses.
  • Operand Stack: A LIFO stack workspace used to calculate execution operations. For example, to calculate 2 + 3, the JVM pushes 2 and 3, executes addition, and pops 5.
  • Frame Data: Contains constants pool resolutions, normal method completion codes, and exception dispatch mappings.

4. Error Scopes: SOE vs. OOM

Property StackOverflowError (SOE) OutOfMemoryError (OOM)
Error Source Call stack depth exceeds limits. Heap allocation space exhausted.
Trigger Reason Infinite recursion or excessively deep method trees. Instantiating objects without freeing references (leak).
Config Flag -Xss<size> (e.g. -Xss1m) -Xmx<size> (e.g. -Xmx2g)

5. JIT Escape Analysis

During runtime compiling optimizations, the JIT analyzer determines if an allocated object escapes a method's namespace:

  • No Escape: If the object stays strictly inside the declaring method (no returns or writes to outer fields), the JIT compiler can optimize this via **Scalar Replacement** (allocating instance fields directly onto the Stack).
  • Escape: If the object is returned, passed to threads, or saved to fields, it escapes and must be instantiated on the Heap.
