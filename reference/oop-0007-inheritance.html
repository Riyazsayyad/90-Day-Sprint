Java Inheritance & Dynamic Dispatch Reference

Study Reference for Initialization Order, Overriding Contracts, and Dispatch Mechanics

1. Object Initialization Sequence

When executing new Child(), the JVM processes class initialization in this exact order:


   +---------------------------------------------+
   |  1. Metaspace Check / Class Loading         | (Recursively loads parent and child classes)
   +----------------------+----------------------+
                          |
                          v
   +----------------------+----------------------+
   |  2. Heap Memory Allocation & Zeroing        | (Allocates combined memory; zeros all fields)
   +----------------------+----------------------+
                          |
                          v
   +----------------------+----------------------+
   |  3. Parent Instance Variables & Init Blocks | (Initializes parent-specific instance fields)
   +----------------------+----------------------+
                          |
                          v
   +----------------------+----------------------+
   |  4. Parent Constructor Execution            | (Parent constructor body executes)
   +----------------------+----------------------+
                          |
                          v
   +----------------------+----------------------+
   |  5. Child Instance Variables & Init Blocks  | (Initializes child-specific instance fields)
   +----------------------+----------------------+
                          |
                          v
   +----------------------+----------------------+
   |  6. Child Constructor Execution             | (Child constructor body executes)
   +----------------------+----------------------+

2. Overriding vs. Overloading vs. Hiding

Property Method Overriding Method Overloading Method Hiding (Static)
Scope Parent and Child classes. Same class (or subclasses). Parent and Child classes.
Signature Must be identical. Name matches, arguments differ. Must be identical.
Resolution Runtime (dynamic dispatch). Compile time (static typing). Compile time (reference type).
Access modifier Cannot be more restrictive. Can be anything. Cannot be more restrictive.

3. Overriding Verification Cheat Sheet

Before overriding a method, verify that the subclass signature compiles against these rules:

  • Return Type: Must be identical, or covariant (e.g. parent returns Number; child returns Integer).
  • Exception throws list: Overridden method throws cannot expand parent throws. 
    // Parent Method declaration
protected void writeData() throws IOException;

// VALID CHILD OVERRIDES
public void writeData(); // Throws nothing
public void writeData() throws FileNotFoundException; // Narrower exception

// INVALID CHILD OVERRIDES
public void writeData() throws Exception; // Broader checked exception
public void writeData() throws SQLException; // New unrelated checked exception
  • Visibility: Subclass method access modifiers must be equal to or broader than parent (e.g., if parent is protected, child must be protected or public).

4. Constructor Chaining Code Template

public class Parent {
    private final String type;

    // Parameterized constructor (No default constructor exists)
    public Parent(String type) {
        this.type = type;
    }
}

public class Child extends Parent {
    private final int value;

    // Subclass constructor must chain explicitly
    public Child(String type, int value) {
        super(type); // Must be the FIRST statement
        this.value = value;
    }
}

5. Design Guidelines

  • Liskov Substitution Principle: Subclasses must compile and behave exactly like their parent type. If client code needs to write instanceof Child checks, your hierarchy violates LSP.
  • Favor Composition over Inheritance: Inherited structures are fragile because changes to a parent class can break subclass internal invariants. Use inheritance only when a true IS-A relationship exists.
  • Design for Extension or Prohibit It: Declare classes final to prevent extensions unless you specifically outline extension boundaries. Keep overridable methods minimum.
  • Avoid calling overridable methods in constructors: Doing so exposes uninitialized child subclass state during initialization, causing NullPointerExceptions.
