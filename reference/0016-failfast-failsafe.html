Fail-Fast vs. Fail-Safe Iteration Reference

Study Reference for Concurrent Traversals

1. What is ConcurrentModificationException (CME)?

CME is thrown when a thread modifies the structural layout of a collection (adding, removing, or clearing elements) while another thread (or the same thread) is traversing it using a fail-fast iterator, unless that modification is performed through the iterator's own methods.

2. Compare & Contrast

Feature Fail-Fast Iterators Fail-Safe / Weakly Consistent
Target Source Operates directly on the backing collection's raw array/node structure. Operates on a cloned array (snapshot) or reads bucket pointers directly.
CME Danger? **Yes**. Throws ConcurrentModificationException if modCount changes. **No**. Guaranteed to never throw ConcurrentModificationException.
State Visibility Immediate. Any change throws an exception or updates the view. Stale snapshot (CopyOnWriteArrayList) or weakly consistent (ConcurrentHashMap).
Memory Overhead O(1) auxiliary space. Very light. Can be O(n) (for copying arrays during write operations).
Examples ArrayList, LinkedList, HashMap, HashSet. CopyOnWriteArrayList, ConcurrentHashMap, CopyOnWriteArraySet.

3. Fail-Fast Check Formula

Every call to next() or remove() inside fail-fast iterators triggers a state validation step:

final void checkForComodification() {
    if (modCount != expectedModCount)
        throw new ConcurrentModificationException();
}

Note: Fail-fast checks are **best-effort** and **not guaranteed** to detect modifications in multithreaded environments due to unsynchronized memory reads/writes. If thread A modifies the collection and thread B iterates, thread B might read a stale cached value of modCount and traverse successfully without throwing an exception, leading to corrupted data structures.

4. CopyOnWriteArrayList Snapshot Iteration

When you modify a CopyOnWriteArrayList, it does not lock the list for reads. Instead, it copies the underlying array:

// Under the hood of CopyOnWriteArrayList.add(E e)
synchronized (lock) {
    Object[] es = getArray();
    int len = es.length;
    Object[] newElements = Arrays.copyOf(es, len + 1);
    newElements[len] = e;
    setArray(newElements); // Swaps reference
}

The iterator keeps a reference to the **original array reference** captured during creation (the snapshot). Because this snapshot array is never mutated, the iterator is 100% immune to ConcurrentModificationException.

5. ConcurrentHashMap Weakly Consistent Iteration

Unlike CopyOnWriteArrayList, ConcurrentHashMap's iterator does not copy arrays. It traverses the actual hash bucket nodes directly.
It is **weakly consistent**:

  • It uses volatile bucket reads to safely traverse existing nodes.
  • It may or may not reflect insertions/updates made to the map after the iterator was created.
  • It is guaranteed to never throw ConcurrentModificationException.
