Object-Oriented Programming (OOP) Paradigm Reference

Paradigm Comparison & Java Specifications

1. Procedural vs. Object-Oriented Programming

Software development paradigms structure code execution and state management differently. The table below highlights key operational contrasts:

Dimension Procedural Programming Object-Oriented Programming
Focus Algorithms, procedures, and sequential functions. Data elements encapsulated with their associated behaviors (Objects).
State Access Global or shared state; exposed directly to functions. Strictly controlled via access modifiers (Encapsulation).
Coupling High. Small changes in data formats break multiple functions. Low. Implementation details are hidden behind class abstractions.
Extension Requires modifying existing conditionals and code blocks. Achieved via Polymorphism and dynamic dispatch (Open/Closed).

2. The Four Pillars of OOP

  • Encapsulation: The wrapping of data (fields) and code (methods) into a single unit. Data is kept private, and access is allowed only through public getters/setters, safeguarding the object's internal state.
  • Abstraction: Exposing only the essential interfaces of an object while hiding implementation details. In Java, this is realized using interface and abstract class structures.
  • Inheritance: Reusing the properties and behaviors of a parent class in a child class. Modeled via the extends keyword. Inherited systems are often tightly coupled, leading to the design preference: Favor Composition over Inheritance.
  • Polymorphism: The ability of a variable, function, or object to take on multiple forms.
    • Compile-time Polymorphism: Method overloading (same method name, different signatures).
    • Runtime Polymorphism: Method overriding and Dynamic Dispatch, where the JVM resolves the exact method implementation at runtime based on the actual object type, not the reference type.

3. Why Java is Not a "Pure" OOP Language

A pure OOP language requires every data type to be an object. Java breaks this purity requirement to optimize execution speeds on the CPU:

  • Primitive Types: Java retains eight primitive types: byte, short, int, long, float, double, boolean, and char. Primitives hold values directly in memory (stack) rather than allocating wrapper envelopes on the Heap.
  • Static Methods & Variables: Static elements belong to the class namespace itself, rather than any instantiated object, allowing execution outside of object contexts.

4. The Java Class Hierarchy Root: java.lang.Object

All Java classes implicitly extend java.lang.Object. This provides a baseline set of behaviors for every object instance in the system:

Method Purpose & Contract
boolean equals(Object obj) Indicates whether some other object is "equal to" this one. Must be overridden alongside hashCode().
int hashCode() Returns a hash code value for the object, crucial for hash-based data structures (like HashMap).
String toString() Returns a string representation of the object, defaulting to classname@hashcode.
Class<?> getClass() Returns the runtime class representation, supporting reflection.
