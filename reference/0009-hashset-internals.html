Oracle HashSet Javadoc

The Backing HashMap

Inside Java's HashSet, elements are stored as keys in a backing HashMap instance. The map values are all bound to a single dummy sentinel object called PRESENT:

public class HashSet<E> extends AbstractSet<E> implements Set<E>, Cloneable, java.io.Serializable {
    private transient HashMap<E,Object> map;

    // Dummy value to associate with an Object in the backing Map
    private static final Object PRESENT = new Object();

    // Default Constructor initializes the backing HashMap
    public HashSet() {
        map = new HashMap<>();
    }
}

Core Delegation Methods

Because the backing map does all the heavy lifting, HashSet's primary methods are simple wrapper delegations:

HashSet Method Backing HashMap Call Return Value Mapping
add(E e) map.put(e, PRESENT) Returns true if map.put returns null (key was absent). Returns false if it returns PRESENT (key was duplicate).
remove(Object o) map.remove(o) Returns true if map.remove returns PRESENT (key existed). Returns false if it returns null (key absent).
contains(Object o) map.containsKey(o) Directly returns map.containsKey(o) boolean result.
iterator() map.keySet().iterator() Delegates to the set of keys in the backing map.

The equals() and hashCode() Contract

When an element is queried or added, the backing HashMap locates its bucket using hashCode() and searches the bucket chain or tree using equals(). For custom classes, you must override both:

  • Rule 1 (Consistency): If two objects are equal according to the equals(Object) method, calling hashCode() on each must produce the same integer result.
  • Rule 2 (Distinction): If two objects produce different hash codes, they are guaranteed to be unequal. If they produce the same hash code (a hash collision), they might still be unequal, which requires checking equals().
Broken Contracts:
  • If you override equals() but not hashCode(): Equal objects will produce different hash codes, landing in different buckets. The set will allow duplicates!
  • If you override hashCode() but not equals(): Different objects will map to the same bucket, but the set will compare them using default reference equality (==). Equal values will not be recognized as duplicates!

add(element) O(1) average / O(log n) worst O(1) amortized O(1)
remove(element) O(1) average / O(log n) worst O(n) (due to shifting) O(n) (due to search)

Note: The worst-case O(log n) for HashSet occurs when elements suffer complete hash collisions, leading to Red-Black Tree treeification of map buckets.

When to Use HashSet

  • Use when: Uniqueness is mandatory, lookup performance is critical, and insertion order is irrelevant.
  • Avoid when: You need to maintain insertion order (use LinkedHashSet instead) or sorted order (use TreeSet instead), or memory overhead per element is highly restricted.
