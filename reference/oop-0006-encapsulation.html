Java Encapsulation & Information Hiding Reference

Study Reference for Access Control, Immutable Design, and Defensive Copying Invariants

1. Three Pillars of Boundary Enforcement

Concept Core Objective Java Implementation Mechanism
Encapsulation Group data and related operations into a single logical unit. Classes with bundled fields and member methods.
Information Hiding Prevent clients from depending on unstable internal details. private fields, package-private helpers, unmodifiable lists.
Abstraction Hide complexity by providing simplified model views. interface declarations and abstract classes (e.g. List vs ArrayList).

2. Defensive Copying & Unmodifiable Wrappers

To avoid representation exposure of internal lists or arrays, follow these patterns:

// Pattern A: Defensive Copying (Constructor & Setter Inputs)
public class Order {
    private final List<LineItem> items;

    public Order(List<LineItem> items) {
        // Create a new list copy to break external references
        this.items = items == null ? new ArrayList<>() : new ArrayList<>(items);
    }
}

// Pattern B: Read-Only Views (Getters)
public class Order {
    private final List<LineItem> items = new ArrayList<>();

    public List<LineItem> getItems() {
        // Return an unmodifiable view (O(1) wrapper overhead)
        return Collections.unmodifiableList(items);
    }
}

// Pattern C: Deep Copying (For collections of mutable elements)
public List<Customer> getCustomers() {
    List<Customer> copy = new ArrayList<>();
    for (Customer c : this.customers) {
        copy.add(new Customer(c.getName(), c.getAddress())); // Copy elements
    }
    return copy;
}

3. How to Construct a Fully Immutable Class

import java.util.ArrayList;
import java.util.Collections;
import java.util.List;

// 1. Declare class final to prevent subclassing mutability
public final class ImmutableReport {
    
    // 2. Make all fields private and final
    private final String title;
    private final List<String> sections;

    // 3. Populate fields only during constructor execution
    public ImmutableReport(String title, List<String> sections) {
        this.title = title;
        // 4. Defensively copy mutable arguments inside the constructor
        this.sections = (sections == null) ? new ArrayList<>() : new ArrayList<>(sections);
    }

    // 5. Provide getters but NO setters
    public String getTitle() {
        return title;
    }

    // 6. Return defensive copy or unmodifiable wrapper in getters
    public List<String> getSections() {
        return Collections.unmodifiableList(sections);
    }
}

4. "Tell, Don't Ask" Rule Book

Anemic/Fragile Approach ("Ask") Encapsulated Approach ("Tell") 
// Client computes discounts manually
double discount = 0.0;
if (cart.getCustomer().isVip()) {
    discount = cart.getSubTotal() * 0.10;
}
double total = cart.getSubTotal() - discount;
 
// Client tells the cart to calculate total
double total = cart.calculateDiscountedTotal();
// (Logic is encapsulated inside ShoppingCart)
 
// Client increments state externally
int score = player.getScore();
score += 10;
player.setScore(score);
 
// Client triggers increment action
player.incrementScore(10);
// (Validations and math kept inside Player)

5. Quick Check Best Practices

  • Minimize Member Visibility: Default to private for fields. If subclasses need read access, prefer package-private or public getters over protected fields, keeping internal state protected.
  • Avoid Dummy Getters/Setters: Never write a getter/setter unless it is actively required. Classes should expose behaviors, not structure.
  • Assert State Invariants: Use Objects.requireNonNull() or check ranges at the boundaries (constructors and public methods) to fail-fast.
  • Copy Collections Eagerly: Never trust collection parameters passed by external callers. Copy them into a local collection immediately.
