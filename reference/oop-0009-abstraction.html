Java Abstraction & API Contracts Reference

Study Reference for Abstraction vs Encapsulation, Interfaces vs Abstract Classes, and API Separation

1. Abstraction vs. Encapsulation

Concept Abstraction Encapsulation
Focus Hiding complexity (what an object does). Hiding data (how an object protects its state).
Level Design level (focus on interfaces/contracts). Implementation level (focus on classes/modifiers).
Enforced by Interfaces, Abstract classes, Public API signatures. private fields, validation logic, final variables.

2. Concrete Class vs. Abstract Class vs. Interface

Feature Concrete Class Abstract Class Interface (Java 8+)
Instantiation Yes (via new). No. No.
Abstract Methods No. Yes (optional). Yes (implicitly abstract).
Instance Variables Yes. Yes. No (only constants).
Multiple Inheritance No. No. Yes, multiple interfaces.

3. Skeletal Implementation Pattern

Combine interfaces and abstract classes to provide maximum API flexibility:

// 1. The Pure behavioral interface
public interface DocumentProcessor {
    void load(String path);
    void extractText();
    void save();
}

// 2. The Abstract Skeletal class containing common logic
public abstract class AbstractDocumentProcessor implements DocumentProcessor {
    protected String path;

    @Override
    public void load(String path) {
        this.path = path; // Common initialization logic
        System.out.println("Document loaded from: " + path);
    }

    // save() and extractText() remain abstract for specific formats
}

// 3. Concrete implementations (Simple and focused)
public class PdfProcessor extends AbstractDocumentProcessor {
    @Override
    public void extractText() {
        System.out.println("Extracting PDF text from: " + path);
    }

    @Override
    public void save() {
        System.out.println("Saving PDF document state");
    }
}

4. Abstraction Best Practices

  • Program to an Interface: Always reference objects by their interface or abstract parent type rather than concrete classes (e.g. List<String> names = new ArrayList<>();).
  • Interface Segregation Principle: Clients should not be forced to implement methods they do not use. Prefer small, highly cohesive interfaces (e.g. Runnable, Comparable) over massive "fat" interfaces.
  • Use Skeleton Abstractions: When designing custom framework modules, always provide a clean interface for maximum client flexibility, paired with an abstract class to provide default helper logic.
  • Prevent Direct Constructor Calls: If a class is only meant to be a base template, declare it abstract even if it doesn't declare any abstract methods.
