Java Abstraction & API Contracts Reference

Study Reference for Abstraction vs Encapsulation, Interfaces vs Abstract Classes, and API Separation

1. Abstraction vs. Encapsulation

Concept Abstraction Encapsulation
Focus Hiding complexity (what an object does). Exposing a simplified model. Hiding data (how an object protects its state). Restricting direct field modifications.
Level Design level (conceptual boundaries). Implementation level (structural restrictions).
Enforced by Interfaces, Abstract classes, Public API signatures, Facades. private fields, validation logic in setters, defensive copying.
Analogy Steering wheel, gas pedal, brake pedal interface. The metal engine casing enclosing pistons, spark plugs, and gears.

2. Type Comparison: Concrete Class vs. Abstract Class vs. Interface

Feature Concrete Class Abstract Class Interface (Java 8/9+)
Instantiation Yes (via new). No. No.
Abstract Methods No. Yes (optional). Yes (implicitly abstract).
Instance Variables Yes. Yes. No (only static final constants).
Constructors Yes. Yes (invoked via super() chaining). No.
Inheritance Single class inheritance. Single class inheritance. Multiple interface inheritance.
Method Bodies All methods must have bodies. Both abstract and concrete methods allowed. Abstract, default, static, and private.

3. Skeletal Implementation Pattern

Joshua Bloch's design strategy from Effective Java combining the flexibility of interfaces with the code-reuse advantages of abstract classes:

// 1. Pure interface contract defining capabilities
public interface DocumentProcessor {
    void load(String path);
    void extractText();
    void save();
}

// 2. Skeletal Abstract class implementing the interface to host shared boilerplate
public abstract class AbstractDocumentProcessor implements DocumentProcessor {
    protected String path;

    @Override
    public void load(String path) {
        this.path = path;
        System.out.println("Common Step: Document loaded from: " + path);
    }

    // extractText() and save() are left abstract for subclass overrides
}

// 3. Concrete subclass extends the skeletal base class
public class PdfProcessor extends AbstractDocumentProcessor {
    @Override
    public void extractText() {
        System.out.println("Reading PDF format at: " + path);
    }

    @Override
    public void save() {
        System.out.println("Writing PDF metadata to disk.");
    }
}

4. Abstraction Best Practices

  • Program to an Interface: Reference variables using parent interface types rather than concrete implementation types (e.g. Map<String, User> map = new HashMap<>();). This permits swapping implementation classes without modifying client code.
  • Interface Segregation Principle (ISP): Break bulky, "fat" interfaces down into smaller, highly cohesive, single-method contracts (like Comparable, Runnable, or AutoCloseable) to avoid forcing clients to implement useless placeholder methods.
  • Avoid Leaky Abstractions: Do not include vendor or implementation-specific details in interface signatures. For example, rather than declaring void write(MySqlRowData row), declare void write(Row row). Avoid throwing low-level database exceptions (SQLException) directly; instead, wrap them in custom runtime exceptions (DataAccessException).
  • Prefer Composition over Inheritance: Inherit interface contracts, but think twice before inheriting large class implementation chains. Use abstract classes mainly as skeletal adapters or templates.
