equals() & hashCode() Cheat Sheet

Core Contracts, Collection Integration, and Production Coding Patterns

1. The equals() Mathematical Contract

For any non-null object references x, y, and z:

Contract Property Requirement Common Pitfalls & Warnings
Reflexive x.equals(x) must return true. Broken by custom collections or floating point comparisons if NaN values are poorly handled.
Symmetric x.equals(y) iff y.equals(x). Highly Vulnerable: Using instanceof across a base class and a subclass. Subclass equals fails to match parent. Enforce strict class checks via getClass().
Transitive If x.equals(y) & y.equals(z), then x.equals(z). Adding a state field in a subclass and comparing polymorphic types will break transitivity if not careful. Favor composition over inheritance.
Consistent Repeated calls return the same outcome unless state is modified. Never perform network-dependent lookups in equals (e.g. java.net.URL.equals() resolves IP addresses over network, breaking consistency and blocking threads!).
Non-nullity x.equals(null) must return false. Ensure null check occurs at the start. getClass() check or instanceof handles null implicitly, but an explicit check is clean.

2. The hashCode() Contract

  1. Consistency during JVM runtime execution.
  2. The Critical Rule: If a.equals(b) == true, then a.hashCode() == b.hashCode().
  3. If a.equals(b) == false, their hashcodes do not have to differ, but different hashcodes optimize table distributions.

4. Production Implementation Checklist

@Override
public boolean equals(Object obj) {
    // 1. Reference check
    if (this == obj) return true;

    // 2. Null and strict type checking (avoids symmetry/subclass violations)
    if (obj == null || getClass() != obj.getClass()) return false;

    // 3. Cast to concrete class
    Employee other = (Employee) obj;

    // 4. Compare fields safely
    return this.id == other.id &&
           Double.compare(this.salary, other.salary) == 0 && // Safe double comparison
           Objects.equals(this.name, other.name);            // Null-safe object reference comparison
}

@Override
public int hashCode() {
    // Combine fields using JDK helper
    return Objects.hash(id, salary, name);
}

5. Special Types & Frameworks Guidelines

Java Records (Java 14+)

  • Records automatically generate a final field for each component and build equals() and hashCode() behind the scenes.
  • Comparing records evaluates all component fields. If you override them inside a record, you risk breaking their data-carrier invariants. Keep records clean.

Hibernate / JPA Entities

  • Database entities have IDs that are null before saving (unpersisted).
  • If you use the ID in hashCode(), the hashcode will change when the entity is saved (violating the Consistency contract).
  • Solution: Use a stable, non-null, unique business key (e.g. email, UUID) if available. If not, use a constant hashcode (e.g. 31) and compare IDs in equals(). Returning a constant forces the entity bucket to a linked list, but Hibernate relies on reference checks or generated keys correctly during updates.

Floating Point Math in equals()

  • Do not use == for floats or doubles inside equals. Use Float.compare(f1, f2) and Double.compare(d1, d2).
  • This handles special values like Float.NaN and handles sign differences between positive and negative zero (-0.0 vs +0.0) correctly.
